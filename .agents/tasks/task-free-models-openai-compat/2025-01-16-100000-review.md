# Free model registry, OpenAI-compatible LLM extractor, and ensemble pipeline

This change adds a full "free/local models" extraction tier to the document intelligence agent. A singleton model registry tracks available OCR, NER, LLM, and layout models with health checks. A new `OpenAICompatExtractor` talks to Ollama, LM Studio, LocalAI, or vLLM via the OpenAI chat completions wire format. An OCR ensemble layer (Tesseract, TrOCR, PaddleOCR, DocTR) and a NER ensemble layer (SpaCy, HuggingFace transformers, LayoutLMv3) feed into an `EnsembleExtractor` that merges results with confidence-weighted scoring. Two new API endpoints (`/extract/local-llm`, `/extract/ensemble`) expose these pipelines. All heavy ML dependencies use try/except imports for graceful degradation.

Watch for: the `OpenAICompatExtractor` constructs a fresh `AsyncOpenAI` client on every extraction call with no connection reuse (confirmed); the NER ensemble's similarity grouping uses character-set Jaccard which incorrectly merges unrelated short entities (likely); the `LayoutLMEngine.extract_with_layout` accepts bounding boxes but ignores them — the pipeline call discards the `bboxes` argument (confirmed); the model registry is a singleton with mutable state but no thread/task safety, and its health-check logic for local models is a shallow import check that doesn't verify the model is actually loaded (confirmed).

## High-level view

The model registry provides a central inventory of available models but currently serves as bookkeeping only — nothing in the extraction pipeline actually consults it before attempting to use an engine. The registration functions (`register_ocr_engines`, `register_ner_engines`) are never called from app startup, and the `EnsembleExtractor` instantiates engines directly regardless of registry status.

The `OpenAICompatExtractor` creates a new `openai.AsyncOpenAI` client per `extract()` call, meaning connection pools are never reused. The retry loop has no delay between attempts — three rapid-fire retries against a struggling endpoint will all fail for the same reason.

The NER ensemble merges results across engines using a character-set Jaccard similarity threshold of 0.8 to group "similar" values. This metric operates on the *set of characters* rather than their sequence, so short strings with overlapping characters (e.g., "Acme" and "Mace" → Jaccard 1.0) will be incorrectly grouped.

The `EnsembleExtractor` coordinates OCR → NER → optional LayoutLM → optional LLM and merges by picking the highest static weight per field. When multiple sources contribute a field, it selects the one with the highest static weight (0.85 LayoutLM > 0.8 NER) rather than comparing actual per-extraction confidence scores — so a low-confidence LayoutLM result always overrides a high-confidence NER result.

The `LayoutLMEngine.extract_with_layout` method accepts bounding boxes but the implementation is identical to text-only processing. The HuggingFace `token-classification` pipeline call never receives positional data.

The orchestrator gains `local_llm` and `ensemble` mode selection. Configuration is cleanly layered: env vars → YAML → provider defaults. Heavy ML deps (`torch`, `transformers`, `paddleocr`) are listed as direct requirements despite being handled as optional imports in code.

4. **NER similarity uses character-set Jaccard** — `_is_similar` computes Jaccard over `set(a_lower)` vs `set(b_lower)`, which is character-set overlap, not sequence similarity. Short entities like "IBM" and "BMI" (Jaccard 1.0) will be incorrectly grouped. Use `SequenceMatcher` or Levenshtein distance instead. (likely)
5. **Model registry is informational only** — The `EnsembleExtractor` builds engines directly without consulting the registry. The registry tracks status that nothing reads. Either wire it into the ensemble's engine selection or document it as future infrastructure. (confirmed)
6. **Ensemble merges on static weight, not actual confidence** — `_merge_all_results` picks the value from the source with the highest static weight (0.85 for LayoutLM, 0.8 for NER, 0.75 for LLM). If NER extracted a field with 0.99 confidence and LayoutLM extracted a different value with 0.3 confidence, LayoutLM wins due to its higher static weight. (confirmed)
7. **TrOCR line splitting fails on multi-column layouts** — The projection-based line detection sums dark pixels per row across the full image width, which merges columns into single "lines" for multi-column documents. (likely)
8. **Heavy ML deps in requirements.txt are not optional** — `torch`, `transformers`, `paddleocr`, `paddlepaddle`, and `python-doctr[torch]` are listed as direct requirements (not in extras). A bare `pip install -r requirements.txt` will download ~5GB of packages even when only Tesseract+SpaCy is needed. Split into `requirements-ml.txt` or use pip extras. (confirmed)

## OpenAI-compatible extractor: client lifecycle and retry behavior

The `_call_endpoint` method creates a fresh `openai.AsyncOpenAI` client every time extraction runs:

```python
async def _call_endpoint(self, prompt: str, text: str) -> str:
    import openai
    client = openai.AsyncOpenAI(
        base_url=self._base_url,
        api_key=self._api_key,
        timeout=self._timeout_seconds,
    )
```

Each client opens its own HTTP connection pool. For batch processing or repeated calls, this means no connection reuse and repeated TLS handshakes (if the endpoint uses HTTPS). The existing `APIExtractor` stores its client at the instance level — this extractor should follow that pattern.

The retry loop runs `self._max_retries` attempts with zero delay:

```python
for attempt in range(self._max_retries):
    try:
        response = await client.chat.completions.create(...)
        return content
    except Exception as e:
        last_error = e
        if attempt < self._max_retries - 1:
            continue
```

Against a local Ollama instance under memory pressure, three rapid-fire retries will all fail for the same reason. Even `await asyncio.sleep(2 ** attempt)` would substantially improve recovery.

## LayoutLM bounding box parameter is dead code

`LayoutLMEngine.extract_with_layout` declares the interface for layout-aware extraction but the implementation is identical to `extract_entities`:

```python
def extract_with_layout(self, text: str, bboxes: List[List[int]]) -> ...:
    ...
    results = self._pipeline(text)  # bboxes never passed
```

LayoutLMv3 requires a feature extractor that takes both tokens and boxes. The current implementation runs text-only token classification regardless of whether bounding boxes are provided. Code that calls `extract_with_layout` with real OCR bounding boxes gets no positional benefit — the method is functionally a duplicate of `extract_entities`.

## NER ensemble similarity grouping

The `_is_similar` method uses character-set Jaccard similarity:

```python
set_a = set(a_lower)
set_b = set(b_lower)
intersection = len(set_a & set_b)
union = len(set_a | set_b)
similarity = intersection / union
return similarity > 0.8
```

This operates on *unique characters*, not their order or frequency. "Acme" and "Mace" have identical character sets (Jaccard = 1.0). Short entity values common in NER (company tickers, abbreviations, short names) are especially vulnerable to false grouping. The OCR ensemble already uses `SequenceMatcher` for text comparison — the NER ensemble should use an equivalent sequence-aware metric.

## Ensemble confidence merging picks static weight over actual extraction quality

The ensemble merge assigns fixed weights per source type:

```python
all_entities.append((ner_entities, 0.8, "ner_ensemble"))
all_entities.append((layoutlm_entities, 0.85, "layoutlm"))
all_entities.append((llm_entities, 0.75, "local_llm"))
```

Then selects the maximum-weight source per field:

```python
best = max(values, key=lambda x: x[1])  # x[1] is the static weight
merged[field] = best[0]
```

LayoutLM always overrides NER for any field both extract, regardless of actual per-field confidence. The NER ensemble might have high agreement (3 engines reporting "Acme Corp" at 0.95+) while LayoutLM produces a garbled token, but LayoutLM's 0.85 static weight wins over NER's 0.8. A better merge would factor in the actual per-value confidence scores that each engine already computes.

## Prompt construction gap when entity config is missing

The `_build_prompt` method falls back to "Extract all relevant entities" when `entities.yaml` lacks a key matching the document type and also lacks a "general" key. This vague instruction may produce inconsistent schemas across calls. If the `/extract/local-llm` endpoint is called without a `document_type` query param, `doc_type` defaults to "general" — which may or may not exist in the YAML. (possible)

## Test coverage gaps

Not tested: the two new API endpoints (`/extract/local-llm`, `/extract/ensemble`) have no integration tests; the ensemble's interaction with real OCR output text feeding into real NER is never exercised end-to-end; the `extract_with_layout` method's test validates the current text-only behavior (confirming the bug rather than the intended layout-aware behavior).
