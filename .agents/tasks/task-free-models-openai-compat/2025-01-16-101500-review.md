# Free model registry and ensemble pipeline — fix verification (v2)

The team addressed all 8 issues from the v1 review. The `AsyncOpenAI` client is now instantiated once in `__init__` for connection reuse. The retry loop sleeps with `2^attempt` exponential backoff. `LayoutLMEngine.extract_with_layout` carries a detailed docstring documenting that bounding boxes are accepted for interface compatibility but not yet wired to the model. The NER ensemble's `_is_similar` now uses `difflib.SequenceMatcher` for sequence-aware comparison. The model registry class docstring explicitly states it is informational and not consulted by the ensemble pipeline. The `_merge_all_results` method now computes `actual_confidence * static_weight` as the effective score, so a high-confidence NER result can beat a low-confidence LayoutLM result. TrOCR's `_split_into_lines` documents its single-column limitation. Requirements are split into core `requirements.txt` and optional `requirements-ml.txt`.

Watch for: the README installation instructions don't mention `requirements-ml.txt`, so users who want ensemble features will miss the heavy ML deps (confirmed); the test for confidence-weighted merging only exercises the case where LayoutLM has *both* higher weight and higher confidence — the critical edge case (NER high-confidence overriding LayoutLM low-confidence) is untested (confirmed); the `_build_confidence_report` in `EnsembleExtractor` still reports per-field confidence based on static source weights rather than the effective scores used during merge, so the reported confidence may not match how the value was actually selected (likely).

**Verdict**: APPROVED

## High-level view

All eight v1 issues are resolved. The fixes are mechanically correct and the code is cleaner for them. The client lifecycle fix eliminates per-call connection churn. The backoff fix prevents rapid-fire retries. The SequenceMatcher swap eliminates false grouping of short strings with overlapping characters. The confidence-weighted merge formula (`actual_confidence × static_weight`) is the right approach — it preserves source-type priors while letting per-extraction quality dominate.

Three minor gaps remain. The README's quickstart shows only `pip install -r requirements.txt` with no mention of `requirements-ml.txt` for users who want TrOCR, PaddleOCR, or LayoutLM. The ensemble merge test suite validates the common path but doesn't exercise the scenario the fix was designed for (high-confidence NER beating low-confidence LayoutLM). The confidence report still derives base confidence from static weights, creating a reporting inconsistency — the value selected by effective score 0.792 (NER: 0.99 × 0.8) might be reported with base confidence 0.85 (LayoutLM's static weight) if LayoutLM also extracted the field.

## Confidence report vs. merge selection inconsistency

The merge selects the winning value per field using:

```python
effective_score = actual_confidence * static_weight
best = max(values, key=lambda x: x[1])  # x[1] is effective_score
```

But the confidence report doesn't know which source won. It reports:

```python
base_confidence = max(w for w, _ in sources_with_field)
```

This takes the highest *static weight* among all sources that contributed the field, regardless of which source's value was actually selected. If both NER (weight 0.8) and LayoutLM (weight 0.85) extracted `vendor`, the report always uses 0.85 as base confidence — even if the merge picked NER's value because it had higher per-extraction confidence.

The practical impact is limited (confidence is advisory), but it means a downstream consumer comparing reported confidence to the actual selection logic would see inconsistencies. A clean fix would pass the winning effective score per field from `_merge_all_results` back into `_build_confidence_report`.

## Missing test for the confidence-override scenario

The existing merge test:

```python
mock_ner.extract_entities.return_value = {
    "organization_names": [("Acme Corp", 0.7)],  # NER: 0.7 * 0.8 = 0.56
}
mock_layoutlm.extract_entities.return_value = {
    "organization_names": [("Acme Corporation", 0.95)],  # LM: 0.95 * 0.85 = 0.8075
}
# LayoutLM wins — but this doesn't prove the fix works
```

The fix was designed so that NER with confidence 0.99 beats LayoutLM with confidence 0.3:
- NER: 0.99 × 0.8 = 0.792
- LayoutLM: 0.3 × 0.85 = 0.255

Without this test, a regression that reverts to static-weight-only selection would pass the existing suite.
