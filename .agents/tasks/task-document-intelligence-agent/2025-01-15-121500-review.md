# Document Intelligence Agent — fix verification (v2)

The fix commit addresses the top 5 issues from the v1 review: industry modules are wired into the orchestrator, CORS credentials dropped to false, batch processing uses BackgroundTasks, ExtractionMode consolidated to a single definition, and upload size limits enforced at 50MB. The two lower-priority issues (in-memory state, SSRF) remain intentionally unaddressed.

Watch for: the batch background task calls `process_batch` which generates its own internal `job_id`, creating a tracking split where the status endpoint polls one ID and progress updates go to another — the job will appear permanently stuck at 0% (confirmed); uploaded files in the batch flow are never cleaned up after processing completes (confirmed).

**Verdict**: NEEDS_CHANGES

## High-level view

The industry module integration is well-structured: the orchestrator classifies documents when no type hint is provided, runs PII detection unconditionally, dispatches to type-specific analyzers (contract, invoice), and runs compliance for legal documents. The broad `try/except` around the entire analysis block means a failure in one module (say, the classifier) prevents all subsequent modules from running. The `setdefault` pattern for storing results means re-processing a document won't overwrite previously computed analysis — a deliberate idempotency guard.

The batch endpoint now returns immediately with a `job_id` and launches work via `BackgroundTasks`, but the internal `BatchProcessor.process_batch` method creates its own job entry in SQLite with a *different* `job_id`. The route pre-creates a job entry for the external ID, but the actual progress updates (item completed, item failed, mark_complete) all target the internal ID. Polling the external job_id will always show 0 completed items and "running" status.

The size limit implementation reads the full file into memory before checking the length — it enforces a ceiling but doesn't protect against slow uploads holding connections open.

## Batch job_id tracking split

The batch endpoint creates a job_id and registers it with the tracker:

```python
job_id = str(uuid4())
_processor._tracker.create_job(job_id, len(file_paths))
```

Then `_run_batch_in_background` calls:

```python
result = await _processor.process_batch(file_paths=file_paths, mode=mode)
```

Inside `BatchProcessor.process_batch`:

```python
job_id = str(uuid4())  # new, different ID
self._tracker.create_job(job_id, len(file_paths))
```

The processor's internal job_id receives all `update_progress` and `mark_complete` calls. The client-facing job_id sitting in SQLite never transitions past "running" and never increments `completed_items`. The status endpoint queries the client-facing ID and will perpetually return 0% progress.

The fix is either: (a) accept a `job_id` parameter in `process_batch` so it uses the pre-created one, or (b) have the background function bypass `process_batch` and drive the processing loop directly using the existing job_id.

## Batch file cleanup gap

The batch `_run_batch_in_background` function stores the result in `_job_metadata` but never removes the files from `uploads/`. Over time this directory grows without bound. A `finally` block in `_run_batch_in_background` that iterates `file_paths` and removes each would close this gap.

## Industry analysis error isolation

The entire `_run_industry_analysis` body sits inside a single `try/except Exception`. The execution order is: classify → PII detect → type-specific (contract or invoice) → compliance. If the `DocumentClassifier` raises (e.g., import error, unexpected input), the except fires immediately and none of the subsequent modules run. The warning message says "partially failed" but in practice it's all-or-nothing.

Per-module try/except blocks would allow PII detection to succeed even if classification fails, and compliance to run even if the contract analyzer hits an edge case. A bug in any one module currently silences all industry analysis for the document.
