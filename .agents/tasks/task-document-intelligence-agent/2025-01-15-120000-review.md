# Document/Contract Intelligence Agent with tri-modal extraction

A greenfield document intelligence system offering three extraction paths: local NLP (spaCy + regex), cloud LLM (OpenAI/Anthropic), and a hybrid mode that runs local first and escalates low-confidence fields to an API. The orchestrator glues document processors, extractors, industry analyzers, and a rule-based action engine into a FastAPI service. The architecture is well-layered with Pydantic models, YAML-driven configuration, and SQLite for audit and progress state.

Watch for: the `_run_industry_analysis` orchestrator hook is a no-op stub that never calls the implemented industry modules — six fully-built analyzers sit unused (confirmed); uploaded files have no size limits, risking memory exhaustion (confirmed); CORS is misconfigured with wildcard origins + credentials which browsers reject (confirmed); the batch endpoint blocks the HTTP request for the full duration of processing rather than returning a job ID for polling (confirmed).

## High-level view

The extraction pipeline follows a processor → extractor → action pattern. Processors (PDF, image, DOCX, text) normalize documents into a `Document` model, extractors pull entities with per-field confidence scores, and the action engine evaluates YAML-defined rules against results to dispatch webhooks or flags. The hybrid extractor evaluates per-field confidence from local extraction and selectively escalates to the API, merging results field-by-field rather than replacing wholesale.

The industry modules (contract analysis, PII detection, compliance checking, invoice processing, document classification, version comparison) are fully implemented but orphaned — the orchestrator's `_run_industry_analysis` method returns the result unchanged. There's no code path that invokes any of these modules on real documents.

The action engine parses a mini-DSL supporting comparison operators and compound conditions (AND/OR). Its string-based condition evaluation means numeric comparisons fail silently when entity values contain currency symbols or commas (e.g., `"$50,000"` won't parse as a float).

Security posture is minimal: CORS is misconfigured, there's no authentication on any endpoint, no file upload size limit, and webhook/action-rule endpoints accept arbitrary URLs creating SSRF vectors.

## Industry modules are implemented but disconnected

The orchestrator's `_run_industry_analysis` method contains a comment "Industry analysis modules will be implemented in FEAT-003" and returns the result unchanged. Meanwhile, `src/industry/` contains fully implemented modules: `ContractAnalyzer` performs clause extraction, obligation detection, date tracking, and risk scoring; `PIIDetector` scans for SSNs, credit cards, emails, and other PII types with configurable redaction; `ComplianceChecker` evaluates documents against regulatory requirements including GDPR.

None of these are called from the main pipeline. The API has no endpoint that exposes industry analysis either. A document processed through any extraction endpoint will never receive contract analysis or PII scanning regardless of its type. The `contract_analyzer` has a passing test suite confirming it works in isolation — the gap is purely in wiring.

## Upload handling and resource exhaustion

The extraction endpoints (`api/routes/extract.py`) accept file uploads with no size constraint. The file content is read entirely into memory (`await file.read()`) before writing to disk. For a multi-gigabyte file, this exhausts worker memory. The batch endpoint compounds this by accepting multiple files simultaneously.

The `uploads/` directory is never cleaned up in the batch flow. The single-extraction endpoint has a `finally` block that removes temp files, but the batch endpoint saves files, processes them, stores paths in `_job_metadata`, and never deletes them. Over time this directory grows without bound.

## Security surface

The FastAPI app sets `allow_origins=["*"]` with `allow_credentials=True`. Per the CORS spec, browsers reject this combination — a wildcard origin cannot be used with credentials. This is broken CORS rather than open CORS.

There is no authentication or authorization on any endpoint. The action rules endpoint (`POST /actions/rules`) allows anyone to create rules that trigger webhook calls to arbitrary URLs — an SSRF vector. The webhook registration endpoint (`POST /webhooks`) similarly accepts arbitrary target URLs. An attacker can register a webhook pointed at internal services and then trigger extraction events to probe them.

API keys for OpenAI and Anthropic default to empty strings. If unset, the API extractor silently produces empty results rather than failing fast with a configuration error.

## Batch endpoint is synchronous despite async design

`POST /batch/process` calls `await _processor.process_batch(...)` which processes all documents before responding. For a 100-document batch at 2 seconds each, the HTTP request blocks for over 3 minutes. The status and result endpoints exist but are only useful after the batch completes — they can't be polled during processing because the initial request hasn't returned yet.

The fix is to launch the batch as a background task (`BackgroundTasks` or a task queue) and return the `job_id` immediately.

## Action engine condition parsing limitations

The condition DSL splits on ` AND ` and ` OR ` as literal strings: (a) conditions cannot mix AND and OR in a single expression (the first match wins), (b) there's no operator precedence or grouping, (c) numeric comparisons like `amount > 10000` require the entity value to be a bare number — but entities are typically strings like `"$50,000"` or `"USD 10,000"` which fail `float()` conversion silently and evaluate to False. The action rule `notify_finance` (trigger: `amount > 10000`) will never fire against real invoice data without preprocessing.

## Duplicate ExtractionMode enum

`ExtractionMode` is defined in both `src/extractors/base.py` and `src/models/extraction_result.py`. The extractors import from `base.py`; the models, API routes, and tests import from `extraction_result.py`. Currently the hybrid extractor creates results using `ExtractionMode` from `base.py` while the Pydantic model's field type annotation references the one from `extraction_result.py`. Pydantic coerces via string value so this works today, but any code doing `result.extraction_mode is ExtractionMode.HYBRID` across module boundaries will get False.

## In-memory state with no persistence

Two subsystems store state only in process memory:

- **Dynamic action rules** (`api/routes/actions.py`): `_dynamic_rules` dict is module-level. Rules created via the API vanish on restart.
- **Webhook registrations** (`src/utils/webhook_client.py`): `_webhooks` dict on the `WebhookClient` instance. With multiple uvicorn workers, each has its own isolated registry.

Both should be backed by the existing SQLite infrastructure (the audit logger and progress tracker already demonstrate the pattern).

## Test coverage gaps

Not tested: PII detector, compliance checker, document classifier, invoice processor, version comparator, any FastAPI route (no integration tests), webhook dispatch, action engine condition evaluation, OCR engine, PDF processor, image processor, DOCX processor. The 35 tests validate the extraction pipeline core and contract analyzer but leave the industry modules, processors, and API layer untested.
